# Расчёт остатков товаров (М.Видео — тестовое задание)

Консольная программа на **Kotlin**, которая читает CSV-файл со списком складских
операций (поступления и продажи), рассчитывает итоговые остатки по товарам внутри
групп и записывает результат в новый CSV-файл.

---

## Идея решения

Программа построена как линейный конвейер: данные проходят через независимые этапы,
каждый отвечает ровно за одно.

```
CSV-файл ──▶ CsvParser ──▶ List<Operation> ──▶ InventoryService ──▶ List<Stock> ──▶ CsvWriter ──▶ CSV-файл
            (чтение)         (модель)            (расчёт)             (результат)      (запись)
```

Такое разделение (принцип единственной ответственности) даёт:
- каждый компонент тестируется отдельно;
- при смене формата ввода (например, на JSON) меняется только парсер, логика расчёта не трогается;
- при провале теста сразу видно, на каком этаже сломалось.

**Ключевая структура данных** — плоский список `List<Stock>`, где `Stock` — это остаток
одного товара в одной группе. Группировка и ранжирование при продаже выполняются «на лету»
через `filter` + `sortedBy`. Это выбрано ради максимальной наглядности кода (см. раздел
«Замечание по производительности»).

---

## Формат данных

### Входной файл

Каждая непустая строка — одна операция. Разделитель полей — `;`. Тип операции
определяется по количеству полей:

| Тип | Формат | Пример | Смысл |
|-----|--------|--------|-------|
| Поступление | `<группа>;<товар>;<кол-во>` (3 поля) | `G1;apple;10` | +10 шт. товара `apple` в группу `G1` |
| Продажа | `<группа>;<кол-во>` (2 поля) | `G1;12` | продать 12 шт. из группы `G1` |

Пустые строки пропускаются. Строка с другим числом полей или нечисловым количеством
вызывает ошибку с указанием номера строки.

### Выходной файл

Формат: `<группа>;<товар>;<остаток>`. Строки отсортированы по группе, затем по товару.

```
G1;apple;0
G1;banana;3
G1;cherry;3
```

---

## Логика продажи

Продажа списывает товары из группы, начиная с **наивысшего ранга**. Ранг определяется
**лексикографической сортировкой по id товара** (по алфавиту). Если товар высшего ранга
закончился, а продать нужно ещё — списание каскадно переходит на следующий по рангу товар.

Если продажа больше суммарного остатка группы — списывается всё, что есть, а разница
записывается с **отрицательным** значением на последний по рангу товар.

**Пример.** Группа `G1`: `apple=10, banana=5, cherry=3`. Продажа `G1;12`:
`apple` (высший ранг) → 0 (списано 10, осталось продать 2), `banana` → 3 (списано 2),
`cherry` не тронут. Результат: `apple=0, banana=3, cherry=3`.

## Как запустить

Пути к файлам определяются по приоритету:
**аргументы командной строки → переменные окружения → значения по умолчанию**
(`input.csv` → `output.csv`).

### Через Gradle (рекомендуется)

```bash
# пути по умолчанию: input.csv -> output.csv
./gradlew run

# с явными путями
./gradlew run --args="input.csv output.csv"
```

> Если файлов Wrapper (`gradlew`) нет в репозитории, сгенерируйте их один раз
> установленным Gradle: `gradle wrapper`. Либо запускайте задачи напрямую: `gradle run`.

### Через переменные окружения

```bash
INPUT_FILE=data/in.csv OUTPUT_FILE=data/out.csv ./gradlew run
```

### Через исполняемый JAR

```bash
./gradlew build
java -jar build/libs/mvideo-inventory-1.0.0.jar input.csv output.csv
```

## Как запустить тесты

```bash
./gradlew test
```

Отчёт о прохождении откроется в консоли. HTML-отчёт после запуска лежит в
`build/reports/tests/test/index.html`.

**Что покрыто (7 тестов в 1 файлах):**

- **`InventoryServiceTest`** — логика остатков: каскадное списание, отрицательный
  остаток при нехватке, накопление при повторном поступлении, чередование операций,
  независимость групп, продажа по пустой группе.

## Что и где смотреть

Если разбираете код — вот маршрут по важности:

1. **`service/InventoryService.kt`** — главное. Здесь вся бизнес-логика: метод `sale()`
   реализует каскадное списание по рангу и обработку нехватки (отрицательный остаток).
2. **`model/Operation.kt`** — как моделируются два типа операций через `sealed class`
   (что даёт проверку полноты `when` на компиляции).
3. **`parser/CsvParser.kt`** — как строка CSV превращается в операцию (различение типов
   по числу полей) и как обрабатываются ошибки.
4. **`writer/CsvWriter.kt`** — записывает результаты остатков в файл.
5. **`InventoryServiceTest.kt`** — тесты логики читаются как спецификация поведения:
   по ним видно, какие сценарии учтены.

---

## Логирование

Используется SLF4J с реализацией Logback. Уровень по умолчанию — `INFO`
(старт, число операций, запись результата, предупреждения о нехватке). Чтобы видеть
каждое поступление и списание, установите уровень логгера `com.example` в `DEBUG`
в `src/main/resources/logback.xml`.
